## 🔴 P0 — 上线前必须修(服务现在根本不可用)

### P0-1 检测器 save/load/score 契约断裂

这是这次最关键的一个 bug。训练侧保存的是"裸 sklearn 模型",推理侧却按"封装类"去调用,导致四种检测器全军覆没。

**实测结论(我把四种都真跑了一遍):**

| 检测器 | 训练/保存 | `/detect` 单条 | `/detect/batch` 批量 |
|--------|----------|----------------|---------------------|
| `gmm`(默认) | ✅ 能存 | ⚠️ 不崩,但**任意输入输出恒定**(见 P0-3) | ❌ 500 崩溃(见 P0-2) |
| `ocsvm` | ✅ 能存 | ❌ 500 `AttributeError` | ❌ 500 |
| `kde` | ❌ 保存即崩 | — | — |
| `deepsvd` | ❌ 保存即崩 | — | — |

**位置:**
- `service/train_service.py:238-239` —— 保存时传的是 `scaler=detector.scaler, detector=detector.model`(裸模型)
- `service/engine.py:144-147` —— 推理时 `self._detector.score(prdc_scaled)`
- `service/detectors.py:61` —— 训练阈值标定用的是封装类 `GMMDetector.score()`,它返回的是 `-score_samples`(每样本负对数似然)

**证据(🔬 实测):**
- `GaussianMixture.score(X)` 返回的是**一个标量**(整批平均对数似然),不是 per-sample,**而且没取负号**。训练时阈值用的是 `-score_samples`(per-sample),两边方向和形状都不一致 → `raw_score >= threshold` 判决无意义。
- `OneClassSVM` **根本没有 `.score()` 方法** → 推理直接 `AttributeError`。
- `KDETector` / `DeepSVDDDetector` **没有 `self.model` 属性**(分别叫 `kde_model` / `encoder`)→ `save_model(detector=detector.model)` 在**保存阶段**就 `AttributeError`,模型根本练不出来。

**根因:** 保存的对象(裸模型)和推理调用的接口(封装类的 `.score()`)不是同一个东西。封装类把"打分语义"(取负、per-sample)收在 `GMMDetector.score()` 里,但保存时把这层壳丢了。

**怎么修:** 让 save / load / score 走**统一契约**。最简单的做法是**保存整个封装类**(它内部已经持有 scaler + model + 打分逻辑),推理时只认 `wrapper.score()`:

```python
# train_service.py —— 存整个 wrapper,而不是拆出 .model
save_model(..., detector=detector, ...)        # detector 是 GMMDetector 实例

# engine.py —— 统一只调 wrapper.score(),它对每种检测器都返回 per-sample 异常分
raw_scores = self._detector.score(prdc)        # 不再自己 transform + 调裸模型
```

并给每种检测器补一个 `save→load→score` 的**一致性测试**(见"测试现状"),这类 bug 一次就能堵死。

> 📚 **能学到什么:** "保存什么"和"加载后怎么用"必须对称。约定好序列化边界(存 wrapper 还是存裸模型),并用 round-trip 测试钉死。另外,**任何代码写完要至少端到端手跑一遍**——这四个检测器没一个真正 work,说明从没完整跑过一次完整链路。

### P0-2 批量接口必崩(两个独立 bug,修了一个还会撞第二个)

`POST /api/v1/detect/batch` 对默认 GMM **每次调用都 500**。这条路径上叠了两个独立缺陷:

**Bug A — `enumerate` 标量(先触发):**
- 位置:`service/engine.py:251` `for i, score in enumerate(raw_scores)`
- 因为 P0-1,GMM 的 `_score` 返回的是 0 维标量。🔬 实测 `enumerate(np.float64(...))` → `TypeError: 'numpy.float64' object is not iterable`。

**Bug B — 响应构造删掉了必填字段(修好 A 之后才会暴露,Codex 抓到):**
- 位置:`service/api.py:196` 显式 `if k != "raw_score"` 把 `raw_score` 过滤掉了
- 但 `service/schemas.py:86` `BatchItemResponse.raw_score` 是**必填**的
- 🔬 实测:`BatchItemResponse(**{不含 raw_score})` → `ValidationError: raw_score Field required`

**怎么修:** A 随 P0-1 一起修(打分返回 per-sample 数组);B 直接别过滤 `raw_score`(或给 schema 设默认值)。补一个 **FastAPI 层**的 batch 端点测试(用 `TestClient`),光修不测下次还会回归。

> 📚 **能学到什么:** schema 的必填字段和实际构造要对齐;同一条路径可能藏多个 bug,修第一个崩溃点后要继续往下走完。

### P0-3 单条检测对任意输入都输出恒定

`POST /api/v1/detect` 单条接口,**不管输入什么日志,算出来的 PRDC 特征恒为 `[1, 0, 1, 1]`**(Precision, Recall, Density, Coverage),于是分数、`is_fault`、`confidence` 全都一样。这个"检测器"等于一个常数函数。

**位置 / 根因:**
- `service/prdc.py:46` `k = min(k, n_query)` —— 单条时 `n_query=1`,把 k 夹成 1(训练时是 5)
- `service/prdc.py:60-64` —— 单点的 query 内部 k-NN 半径 `r_query=0` → **Recall 恒 0**
- `service/prdc.py:98` `density = density / (density.max() + 1e-8)` —— 单条时自己除自己 → **Density 恒 ≈1.0**

**证据(🔬 实测):** 连续喂 5 条不同的随机查询,density 全是 `1.000000`;单条 PRDC 全是 `[1, 0, 1, 1]`。

**怎么修:** PRDC 的推理态公式需要重新设计(详见 P0-4),核心是:**不要把"只依赖当前这一批 query 的统计量"塞进特征**,并且 `k` 不应被 `n_query` 夹小(只有"query 内部半径"那一步才受 query 数量限制)。

> 📚 **能学到什么:** 上线前哪怕造两条假数据各调一次,就能立刻发现"输出永远一样"。**自测要覆盖'不同输入应给不同输出'这种最基本的性质。**

### P0-4 PRDC 训练/推理分布不一致 + 每请求重算 O(n_ref²)

**问题一 —— 训练/推理不是同一个特征空间(train-inference skew):**
训练时 `compute_prdc(train, train)`(query 就是 ref,整批一起算),推理时 `compute_prdc(normal_ref, query)`(固定参考集 + 流式查询),两者语义不同:
- 🔬 实测同一条日志放进不同 batch,**density 在 0.78↔0.95、recall 在 33.6↔3.2 之间跳变**(recall 居然 >1,说明公式在 `n_ref ≫ n_query` 时根本不成立)。
- density 训练时是**跨整个训练集**归一化(分布在 0.68~1.0),推理时按**当前 batch** 归一化 → StandardScaler 标定的尺度和推理时对不上,喂给检测器的特征不在同一个空间。
- 后果:批量结果会**随同一批里的其它日志而变**;单条和批量对同一条日志给不同答案。

**问题二 —— 性能/OOM/DoS:**
- 位置:`service/prdc.py:48` `torch.cdist(E_ref, E_ref)`
- 这个 `n_ref × n_ref` 距离矩阵**每次请求都重算一遍**,但它只依赖固定的参考集。n_ref=1万 → 每个请求瞬时分配 ~400MB 矩阵。**既是性能灾难,也是一条 OOM/DoS 入口**(一条请求就能打爆显存/内存)。而且 PRDC 跑在 `_embed_lock` 之外,并发请求会同时分配多个大矩阵。

**怎么修:**
1. **把只依赖参考集的量在 `load()` 时算一次并缓存**:`r_ref`(每个 ref 点的 k-NN 半径)、`mean_ref_radius`、以及训练时 density 的归一化常数。请求时只算 `query → ref` 的距离。
2. **去掉 query-set 相对的归一化**(`density.max()`),改用训练时存下来的固定常数。
3. `k` 不要被 `n_query` 夹小(只有"query 内部半径"那一步需要 `min(k+1, n_query)`)。
4. 重新确认单条(n_query=1)场景下 Recall / Coverage 的定义是否还成立。

> 📚 **能学到什么:** 这是机器学习工程最经典的坑——**训练和推理的特征必须同分布**。任何依赖"当前这批数据"的统计量(max / mean / 分位数)都不能直接进特征;固定参考集的统计应在训练时算好存下,推理复用。顺带:重活只算一次,别放进每个请求的热路径。

## 🟠 P1 — 安全与稳定性

### P1-1 请求里的 `model_version` 被静默忽略
请求 schema(`schemas.py:23`)和 README 都说能指定模型版本,但 `detect_single` / `detect_batch`(`api.py:166`)压根不收这个参数,永远用启动时加载的那个模型。`_resolve_version`(`api.py:112`)定义了却**从没被调用**;`model_info` 也只认当前版本,其它一律 404。**宣传的"多模型"是假的。**
**修:** 要么真做多模型缓存,要么请求传了非默认版本就明确返回 400/404,别静默吞掉。

### P1-2 健康检查不反映就绪状态
默认模型不存在时,启动只打一行 warning(`api.py:54`),但 `/health` 照样返回 `"healthy"`(`api.py:118`,`degraded`/`unhealthy` 状态从没被用过);此时真去调 detect 会在引擎未加载时抛 `RuntimeError`(`engine.py:184`)→ 裸 **500 而不是干净的 503**。
**修:** 无模型时 health 返回 degraded/unhealthy,检测接口返回明确 503;让健康检查真正反映"能不能干活"。

### P1-3 joblib/pickle 反序列化 = RCE 面
`service/model_loader.py:125,130` 用 `joblib.load(*.pkl)`,底层是 pickle,**加载不可信文件可执行任意代码**。compose 又把宿主 `./models` 挂进容器(`docker-compose.yml:21`),README 还指引"把模型目录丢进 `models/` 就能上线"→ 只要模型目录被污染就有 RCE。
**修:** 模型目录**只读挂载**、收紧写权限、对 artifact 校验 hash/签名;容器别用 root 跑(见 P3)。

### P1-4 CORS `*` + 凭证 + 无鉴权 + 监听 0.0.0.0
`api.py:92-98` `allow_origins=["*"]` + `allow_credentials=True`——Starlette 在这种组合下会**反射任意 Origin 并带上凭证**,等于允许任何网站携带 cookie 跨域调用。叠加**接口零鉴权**、直接 `0.0.0.0` 暴露端口、又没有限流和请求体大小限制,而检测接口本身计算很重(P0-4)→ 很容易被误用或打挂。
**修:** 生产环境收紧 CORS origin;加内部鉴权/API key;反向代理做限流;限制 body 大小。`.env.example` 自己写了 "restrict in production" 却默认 `*`,先把默认值改安全。

### P1-5 `DEVICE=cpu` 被忽略
`service/config.py:97-102` 的 `resolve_device()` 只要 `cuda.is_available()` 就返回 `"cuda"`,**完全无视用户在 `.env` 里设的 `DEVICE=cpu`**(和 README"修改默认设备"自相矛盾)。而真正尊重配置的 `get_device()`(`config.py:50`)是**死代码,从没被调用**。
**修:** 删掉重复实现,统一用一个尊重 `Settings.DEVICE` 的函数,CUDA 不可用时再回退 CPU。

### P1-6 依赖不锁版本(本机已复现装崩)
`requirements.txt` 全用 `>=`(注释写着 "Pin major versions" 实际没锁)。🔬 实测后果:`pytest -q` 因缺 `pytest_asyncio` 直接 collection error;尝试导入 schema 又撞上 `pydantic` 与 `pydantic-settings` 版本不兼容(`No module named 'pydantic._internal._signature'`)。`httpx` 还重复列了两次。
**修:** 用 lockfile 或精确 pin 版本,CI 从干净环境安装并跑通测试。

## 🟡 P2 — 部署与健壮性

### P2-1 CPU 镜像健康检查永远失败
`docker-compose.cpu.yml:29` healthcheck 用 `curl`,但 `Dockerfile.cpu` 基于 `python:3.10-slim`(**不含 curl**;只有 GPU 镜像才 `apt install curl`)→ 容器永远 unhealthy,会被编排平台反复重启。
**修:** CPU 镜像里装 curl,或改用 `python -c "import urllib.request,sys; urllib.request.urlopen('http://localhost:8000/health')"` 做探针。

### P2-2 预下载的 embedder 被挂载覆盖
`Dockerfile.gpu:43-48` 构建时把 embedder 预下载进 `/app/models`,但 `docker-compose.yml:21` 又把宿主 `./models` 挂上去**把它盖掉了** → 首次请求仍要联网下载,"预热加速"失效。
**修:** embedder 缓存和训练产物分开放(不同目录),别让 volume 盖掉镜像里预置的缓存。

### P2-3 训练阈值偷工
`train_service.py:207-222` 阈值只取**正常分数的 95 分位**(没用任何异常样本,docstring 里"optimal F1"是空话),且 val 缺失时退化成用 **train 自己的分数**(检测器见过的数据)标定 → 阈值过拟合,实际误报率不可控。
**修:** 至少用独立 val 集;有条件就引入少量带标注异常做真正的阈值标定。

### P2-4 训练脚本 val 检测会崩
`train_service.py:130` `(data_dir / val_file).exists()`,当 `--data-path` 没给(`data_dir=None`)时 `None / "val_normal.jsonl"` → `TypeError`。
**修:** 先判 `data_dir is not None` 再拼路径;这段 `and/or` 混用的逻辑整体重写更清楚。

### P2-5 `.env.example` 的 `CORS_ORIGINS=*` 可能导致启动崩溃
该字段是 `list[str]`,pydantic-settings 默认按 JSON 解析 env 值,`*` 不是合法 JSON。compose 里没传这个变量所以用默认值没事,但谁照着 `.env.example` 生成 `.env` 就可能起不来。**请你们自己验证一下。**
**修:** 注释说明该值要写成 JSON 数组(如 `["https://a.com"]`),或在 Settings 里加自定义解析器支持逗号分隔。

### P2-6 compose 把源码目录挂进容器
`docker-compose.yml:22` 把 `./service` 挂进容器,生产环境会出现"宿主机代码覆盖镜像代码",镜像构建结果和实际运行的不是一回事。
**修:** 这个挂载留给开发用的 compose override,生产 compose 别挂源码。

## 🟢 P3 — 代码质量(不阻塞上线,但建议清理)

- **用 `print` 而不是 `logging`;`LOG_LEVEL`/`HOST`/`PORT`/`RELOAD` 四个配置项定义了从没被用**(`config.py:37-40`,CMD 里硬编码了 uvicorn 参数)。
- 🔬 **`Settings()` 在热路径重复实例化**(`engine.py:125,140` 等共 10 处),每次都重新读一遍 `.env`;明明 `config.py:105` 已有单例 `settings` 不用。
- **三套 device 解析函数**(config 两个 + train_service 一个)语义还不一致;死代码 `_resolve_version` / `get_device`。
- **命名:** `KDETector` 拼写(应为 `KDEDetector`);`deepsvd` 和 `deepsvdd` 不一致。
- 🔬 **`model_version` 字段撞 pydantic 保护命名空间 `model_`**,导入时报 UserWarning(将来 pydantic 收紧可能直接报错)→ 给 schema 设 `model_config['protected_namespaces'] = ()` 或改字段名。
- `requirements.txt` 里 `httpx` 列了两次;无 `.dockerignore`(构建上下文含 `.git`/`models`);容器 **root 运行**;compose `version: "3.9"` 已废弃。

## 测试现状(等于没测)

- 🔬 `pytest -q` 当前**连收集都失败**(缺 `pytest_asyncio`)。
- 测试设计是"**有训练好的模型才跑,没有就 skip**"(`tests/test_service.py:11-19`)→ 上面 P0 那些崩溃路径**一条都没被覆盖到**。
- 没有 FastAPI 层(`TestClient`)的端点测试,所以 P0-2 的批量崩溃、P1-2 的健康检查问题测试都发现不了。

**建议:** 造一个**极小的假模型 fixture**(几条假日志训出来,提交进仓库或在测试里现训),让 CI 在没有真实模型时也能跑通主路径——尤其是 `save→load→score` 一致性、单条/批量端点、health 端点。

---

## 修复建议顺序

1. **P0-1 / P0-2:** 先修模型 save/load/score 契约 + 批量响应,配一个极小训练 fixture 做端到端测试——让服务先能正确跑起来。
2. **P0-3 / P0-4:** 重构 PRDC 推理路径(预算 ref 统计、去掉 batch 相对归一化、修 k 夹断),保证单条/批量/训练三者特征同分布。
3. **P1-1 / P1-2:** 修 model_version 行为和健康检查语义。
4. **P1-6:** 锁依赖,把 CI 跑起来。
5. **P1-3 / P1-4 / P1-5 + P2:** 上线配置加安全边界——非 root、只读模型挂载、收紧 CORS、加鉴权限流、修两个镜像/healthcheck 问题。
6. **P3:** 顺手清理日志、死代码、命名、重复依赖。
